## 1. 액터 정의

| 액터 | 설명 | 주요 목표 |
|------|------|----------|
| **고객 (Customer)** | 상품을 주문하고 배송받는 일반 소비자 | 편리한 장보기, 빠른 배송 |
| **마트 (Store Owner)** | 마트를 운영하고 상품을 판매하는 사장님 | 매출 증대, 효율적 주문 관리 |
| **배달원 (Rider)** | 마트에서 고객까지 상품을 배달하는 사람 | 수익 창출, 효율적 배달 |
| **관리자 (Admin)** | 플랫폼 전체를 운영 관리하는 담당자 | 플랫폼 안정성, 품질 관리 |
| **시스템 (System)** | 자동화된 처리를 수행하는 시스템 | 자동 주문 생성, 알림 발송 |
| **PG사 (Payment Gateway)** | 결제 처리를 담당하는 외부 시스템 | 결제 승인, 환불 처리 |

## 3. 고객 유스케이스

### UC-C01: 회원가입

| 항목 | 내용 |
|------|------|
| **유스케이스 ID** | UC-C01 |
| **유스케이스명** | 회원가입 |
| **액터** | 고객 |
| **개요** | 신규 고객이 동네마켓 서비스에 가입한다 |
| **사전조건** | 고객이 앱/웹에 접속한 상태 |
| **사후조건** | 회원 정보가 저장되고 서비스 이용이 가능해짐 |

**기본 흐름:**
1. 고객이 회원가입 버튼을 클릭한다
2. 시스템이 가입 양식을 표시한다
3. 고객이 이메일, 비밀번호, 이름, 휴대폰 번호를 입력한다
4. 고객이 휴대폰 인증을 요청한다
5. 시스템이 인증번호를 발송한다
6. 고객이 인증번호를 입력한다
7. 시스템이 인증을 확인하고 회원 정보를 저장한다
8. 시스템이 가입 완료 메시지를 표시한다

**대안 흐름:**
- 3a. 이미 가입된 이메일인 경우
  - 시스템이 중복 이메일 오류를 표시한다
  - 고객이 다른 이메일을 입력한다
- 3b. 소셜 로그인으로 가입하는 경우
  - 고객이 카카오/네이버/구글 로그인을 선택한다
  - 외부 인증 후 추가 정보(휴대폰)를 입력한다

**비즈니스 규칙:**
- BR-U01: 이메일과 휴대폰 번호는 시스템 전체에서 유일해야 함

### UC-C02: 로그인

| 항목 | 내용 |
|------|------|
| **유스케이스 ID** | UC-C02 |
| **유스케이스명** | 로그인 |
| **액터** | 고객, 마트, 배달원, 관리자 |
| **개요** | 사용자가 시스템에 로그인한다 |
| **사전조건** | 가입된 회원 |
| **사후조건** | 인증 토큰이 발급되고 서비스 이용 가능 |

**기본 흐름:**
1. 사용자가 이메일과 비밀번호를 입력한다
2. 시스템이 자격 증명을 확인한다
3. 시스템이 JWT 토큰을 발급한다
4. 시스템이 사용자 역할에 맞는 화면을 표시한다

**대안 흐름:**
- 2a. 잘못된 자격 증명인 경우
  - 시스템이 오류 메시지를 표시한다
  - 5회 실패 시 계정을 일시 잠금한다
- 1b. 소셜 로그인을 선택한 경우
  - 외부 OAuth 인증을 수행한다
  - 연동된 계정 정보로 로그인한다

### UC-C03: 주변 마트 검색

| 항목 | 내용 |
|------|------|
| **유스케이스 ID** | UC-C03 |
| **유스케이스명** | 주변 마트 검색 |
| **액터** | 고객 |
| **개요** | 고객이 현재 위치 기반으로 주변 배달 가능 마트를 검색한다 |
| **사전조건** | 고객이 로그인하고 위치 권한을 허용한 상태 |
| **사후조건** | 배달 가능한 마트 목록이 표시됨 |

**기본 흐름:**
1. 고객이 마트 탭을 선택한다
2. 시스템이 고객의 현재 위치를 확인한다
3. 시스템이 배달 반경 내 영업 중인 마트를 조회한다
4. 시스템이 거리순으로 정렬된 마트 목록을 표시한다
5. 각 마트의 이름, 평점, 배달비, 최소주문금액이 표시된다

**대안 흐름:**
- 2a. 위치 권한이 없는 경우
  - 시스템이 배송지 주소 입력을 요청한다
  - 고객이 주소를 입력하면 해당 위치 기준으로 검색한다
- 4a. 필터를 적용하는 경우
  - 고객이 카테고리, 최소주문금액, 배달비 필터를 선택한다
  - 시스템이 필터 조건에 맞는 마트를 표시한다

**비즈니스 규칙:**
- BR-S02: 배달 가능 반경은 최대 5km

### UC-C04: 상품 검색

| 항목 | 내용 |
|------|------|
| **유스케이스 ID** | UC-C04 |
| **유스케이스명** | 상품 검색 |
| **액터** | 고객 |
| **개요** | 고객이 원하는 상품을 검색한다 |
| **사전조건** | 고객이 앱에 접속한 상태 |
| **사후조건** | 검색 조건에 맞는 상품 목록이 표시됨 |

**기본 흐름:**
1. 고객이 검색창에 키워드를 입력한다
2. 시스템이 자동완성 추천어를 표시한다
3. 고객이 검색을 실행한다
4. 시스템이 배달 가능 마트의 상품 중 키워드와 일치하는 상품을 조회한다
5. 시스템이 상품 목록을 표시한다 (상품명, 가격, 마트명, 이미지)

**대안 흐름:**
- 4a. 검색 결과가 없는 경우
  - 시스템이 "검색 결과가 없습니다" 메시지를 표시한다
  - 유사 상품이나 인기 상품을 추천한다
- 5a. 카테고리로 탐색하는 경우
  - 고객이 카테고리를 선택한다
  - 해당 카테고리의 상품 목록이 표시된다

### UC-C05: 장바구니 관리

| 항목 | 내용 |
|------|------|
| **유스케이스 ID** | UC-C05 |
| **유스케이스명** | 장바구니 관리 |
| **액터** | 고객 |
| **개요** | 고객이 장바구니에 상품을 추가/수정/삭제한다 |
| **사전조건** | 고객이 로그인한 상태 |
| **사후조건** | 장바구니 정보가 업데이트됨 |

**기본 흐름:**
1. 고객이 상품 상세 페이지에서 "장바구니 담기"를 클릭한다
2. 시스템이 수량 선택 UI를 표시한다
3. 고객이 수량을 선택하고 확인한다
4. 시스템이 장바구니에 상품을 추가한다
5. 시스템이 장바구니 아이콘에 수량을 표시한다

**대안 흐름:**
- 4a. 다른 마트 상품이 이미 장바구니에 있는 경우
  - 시스템이 "다른 마트 상품이 있습니다. 비우고 담으시겠습니까?" 확인을 요청한다
  - 고객이 확인하면 기존 장바구니를 비우고 새 상품을 추가한다
- 4b. 재고가 부족한 경우
  - 시스템이 최대 구매 가능 수량을 표시한다

**비즈니스 규칙:**
- 장바구니는 마트별로 분리 (한 번에 한 마트에서만 주문 가능)

### UC-C06: 주문하기

| 항목 | 내용 |
|------|------|
| **유스케이스 ID** | UC-C06 |
| **유스케이스명** | 주문하기 |
| **액터** | 고객, PG사 |
| **개요** | 고객이 장바구니 상품을 주문하고 결제한다 |
| **사전조건** | 장바구니에 상품이 있고, 고객이 로그인한 상태 |
| **사후조건** | 주문이 생성되고 결제가 완료됨 |

**기본 흐름:**
1. 고객이 장바구니에서 "주문하기"를 클릭한다
2. 시스템이 주문서 화면을 표시한다
3. 고객이 배송지를 선택/입력한다
4. 고객이 배송 요청사항을 입력한다
5. 고객이 쿠폰/포인트 사용 여부를 선택한다
6. 시스템이 최종 결제 금액을 계산하여 표시한다
7. 고객이 결제 수단을 선택한다
8. 고객이 결제하기 버튼을 클릭한다
9. PG사가 결제를 처리한다
10. 시스템이 주문을 생성하고 마트에 알림을 발송한다
11. 시스템이 주문 완료 화면을 표시한다

**대안 흐름:**
- 6a. 최소 주문 금액 미달인 경우
  - 시스템이 최소 주문 금액 안내 메시지를 표시한다
  - 고객이 상품을 추가한다
- 9a. 결제 실패인 경우
  - 시스템이 결제 실패 메시지를 표시한다
  - 고객이 다른 결제 수단을 선택한다
- 3b. 예약 주문인 경우
  - 고객이 배송 희망 일시를 선택한다 (2시간 후 ~ 7일 이내)

**비즈니스 규칙:**
- BR-O03: 최종 결제 금액 = 상품총액 + 배달비 - 할인 - 포인트
- BR-O04: 예약 주문은 최소 2시간 후 ~ 최대 7일 이내 설정 가능
- BR-P02: 포인트 적립은 결제 금액의 1%

### UC-C07: 주문 취소

| 항목 | 내용 |
|------|------|
| **유스케이스 ID** | UC-C07 |
| **유스케이스명** | 주문 취소 |
| **액터** | 고객, PG사 |
| **개요** | 고객이 주문을 취소한다 |
| **사전조건** | 주문이 마트 접수(ACCEPTED) 전 상태 |
| **사후조건** | 주문이 취소되고 결제가 환불됨 |

**기본 흐름:**
1. 고객이 주문 내역에서 취소할 주문을 선택한다
2. 고객이 "주문 취소" 버튼을 클릭한다
3. 시스템이 취소 사유 선택을 요청한다
4. 고객이 취소 사유를 선택한다
5. 시스템이 취소 가능 여부를 확인한다
6. 시스템이 PG사에 환불을 요청한다
7. PG사가 환불을 처리한다
8. 시스템이 주문 상태를 CANCELLED로 변경한다
9. 시스템이 사용된 포인트/쿠폰을 복원한다
10. 시스템이 취소 완료 알림을 발송한다

**대안 흐름:**
- 5a. 마트가 이미 접수한 경우
  - 시스템이 "마트에서 접수한 주문은 취소할 수 없습니다" 메시지를 표시한다
  - 고객 센터 연결을 안내한다

**비즈니스 규칙:**
- BR-O01: 주문 취소는 마트 접수(ACCEPTED) 전까지만 가능
- BR-P01: 결제 취소는 배송 시작 전까지만 전액 취소 가능

### UC-C08: 실시간 배달 추적

| 항목 | 내용 |
|------|------|
| **유스케이스 ID** | UC-C08 |
| **유스케이스명** | 실시간 배달 추적 |
| **액터** | 고객 |
| **개요** | 고객이 배달원의 실시간 위치와 예상 도착 시간을 확인한다 |

| **액터** | 관리자, 시스템 |
| **개요** | 관리자가 마트/배달원 정산을 관리하고 실행한다 |
| **사전조건** | 정산 대상 주문/배달이 존재함 |
| **사후조건** | 정산이 완료되고 입금됨 |

**기본 흐름:**
1. 시스템이 정산 주기에 따라 정산 데이터를 집계한다

3. 관리자가 정산 내역을 검토한다
4. 관리자가 "정산 실행"을 클릭한다
5. 시스템이 각 대상의 계좌로 입금을 요청한다
6. 시스템이 정산 상태를 COMPLETED로 변경한다

**대안 흐름:**
- 5a. 입금 실패 시
  - 시스템이 정산 상태를 FAILED로 변경한다
  - 관리자에게 알림을 발송한다
